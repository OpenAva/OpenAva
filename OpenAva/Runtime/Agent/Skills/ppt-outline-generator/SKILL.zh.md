# PPT 大纲生成器

使用此技能将主题、零散笔记、会议纪要或粗略草稿转化为清晰的 PPT 大纲。

此技能专注于演示文稿规划，而非最终幻灯片渲染。

## 适用场景

- 从单个主题规划完整的演示文稿。
- 将零散笔记、纪要或长文本整理为清晰的大纲。
- 构建报告、提案、回顾或方案演示文稿的结构。
- 确定每张幻灯片的主要结论和内容重点。

## 边界

- 仅生成大纲和页面级内容设计。
- 不声称能生成可播放的 PPT 或设计文件。
- 不编造业务指标、内部数字或研究结论。
- 若缺少证据，建议应收集哪些数据。

## 默认设置

- 语言：匹配用户语言，不明确时默认简体中文。
- 目标：`信息汇报`。
- 受众：`混合型`。
- 风格：`咨询式`。
- 长度：8 至 12 张幻灯片。
- 输出：仅 `markdown`。

使用默认设置时，在 `假设` 下展示。

## 输入

将请求规范化为以下字段：

- `topic`: 必填的演示主题。
- `objective`: 需要达成的决策、汇报或说服目标。
- `audience`: `management`（管理层）、`business`（业务）、`engineering`（技术）、`mixed`（混合）或 `external`（外部）。
- `duration_or_pages`: 演讲时长或目标幻灯片数量。
- `style`: `consulting`（咨询式）、`minimal_tech`（极简技术型）或 `passionate_report`（热情汇报型）。
- `input_materials`: 笔记、草稿副本、纪要、摘要或来源材料。

若缺少 `topic`，在生成大纲前先提出 1 至 3 个简短澄清问题。

若缺少其他字段：

- 若答案能实质性改善大纲，最多提出 3 个聚焦问题。
- 若用户要求不提后续问题直接生成通用大纲，则使用默认设置并清晰说明假设。

## 工作流程

### 1. 确定故事线

- 识别用户目标、可能的受众和预期的决策或收获。
- 若来源材料较长，在结构化之前先简要总结。
- 选择适合使用场景的叙事方式。

### 2. 构建结构

- 用 1 至 2 句完整句子陈述整个演示的一个顶层 `核心信息`。
- 将其分解为 3 至 5 个一级章节。
- 章节尽量符合 MECE 原则：避免重复，覆盖主要维度。
- 优先采用自然叙事流，除非用户明确要求结论优先的故事结构。

### 3. 设计每张幻灯片

每张幻灯片应包含：

- `page_title`: 直接陈述结论的行动标题。
- `key_message`: 1 至 3 句解释幻灯片要点的句子。
- `supporting_points`: 2 至 5 条支撑要点。
- `evidence`: 可选的证据条目或数据收集建议。
- `visual_suggestion`: 推荐的图表或视觉格式及理由。
- `notes`: 可选的演讲备注。

始终使用行动标题。

好的示例：

- `生成式 AI 已成为工程效率的重要杠杆`
- `交付延误现在集中在测试和发布阶段`

避免中性标题，例如：

- `生成式 AI 使用场景`
- `问题分析`

### 4. 谨慎处理证据

当真实数据不可用时，使用数据收集提示而非编造数字，例如：

- `数据收集建议：从内部 CI/CD 系统中测量过去 12 个月的发布频率和平均构建时间。`

### 5. 适配风格

保持结构在不同风格间稳定，改变语气、重点和视觉偏好。

#### consulting（咨询式）

- 正式、结构化、以决策为导向的语言。
- 优先使用结构化论点和专业图表。

#### minimal_tech（极简技术型）

- 更短的句子和更紧凑的幻灯片密度。
- 聚焦于关键能力、指标和影响。

#### passionate_report（热情汇报型）

- 更有活力、故事驱动的措辞。
- 以成功案例、对比和令人印象深刻的亮点开场。

### 6. 渲染输出

仅以 Markdown 格式渲染结果。

- 使用 `#` 表示演示文稿标题。
- 使用 `##` 表示顶层章节。
- 使用 `###` 表示每张幻灯片标题。
- 使用列表项表示 `supporting_points`。
- 使用带标签的短行表示 `Key Message`、`Evidence`、`Visual Suggestion` 和 `Notes`。
- 在提升可读性时，使用 `---` 分隔幻灯片。

### 7. 回复前自查

根据以下质量标准检查大纲：

- 整个演示文稿有一个清晰的顶层核心信息。
- 有 3 至 5 个一级章节。
- 每张幻灯片聚焦于一个主要结论。
- 标题是行动标题，而非主题标签。
- 章节结构对用户目标足够符合 MECE 原则。
- 所选风格一致体现。
- 缺失证据标注为假设或待收集数据，而非编造事实。

## 输出顺序

始终返回人类可读的 Markdown 结果。

- 如需要，先列出 `假设`
- `顶层核心信息`
- 逐章节的幻灯片大纲
- 可选的 `缺失信息` 或 `建议扩展方向`

## 错误处理

- 若主题缺失，先提简洁澄清问题。
- 若请求超出范围，说明边界并仍然提供大纲。
- 若用户提出相互冲突的目标（如"极简"和"每张幻灯片都要有大量细节"），指出矛盾并提供实际的折中方案。
- 若用户要求其他输出格式，解释此技能返回 Markdown，然后提供 Markdown 大纲。

## 最终规则

显式假设优于隐藏猜测。

目标是产出一个逻辑清晰、易于继续编辑、可安全交给人类设计师或下一个自动化步骤的 PPT 大纲。
