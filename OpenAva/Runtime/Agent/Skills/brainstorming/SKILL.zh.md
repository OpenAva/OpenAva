# 将想法头脑风暴为设计方案

使用此技能将原始想法转化为清晰的、经过确认的方案，再开始执行。

## 硬性关卡

在以下条件满足之前，不得开始执行操作：

1. 已提出具体的建议或方案
2. 用户已确认下一步行动

对于非常小、低风险的任务，一个简洁的建议就足够了（不需要完整的方案文档）。

## 存在意义

"简单"的任务仍会因隐藏假设而失败。好的头脑风暴通过提出更少、更好的问题来减少走弯路。

## 自适应深度指导

根据复杂度和不确定性调整交互深度：

- 低复杂度：简洁澄清，直接给出建议，一次确认
- 中等复杂度：简要列出备选方案和权衡
- 高不确定性：更完整的方案细节、书面文档和更深入的自我审查

不要向用户明确标注模式；无缝调整深度即可。

## 工作流程清单（按顺序完成）

1. 探索项目背景（相关文件、文档、近期变更）
2. 只提问高影响力的澄清性问题
3. 生成备选方案（数量根据复杂度调整）并按决策矩阵排名
4. 先给出一句话建议，再简要说明理由
5. 与用户确认下一步行动
6. 若不确定性/影响较高，编写方案文档并进行更深入的自我审查
7. 获得批准后，过渡到执行阶段

## 流程规则

### 1) 先探索背景

- 在提出变更建议之前，先检查现有背景和模式
- 保持范围现实可行
- 若请求过大，拆分为子项目，仅规划第一个

### 2) 仅通过关键问题澄清

- 只有在缺失信息会实质性改变方案时才提问
- 严格控制问题数量：规划阶段默认 0 至 3 个问题
- 优先关注目标、硬性约束和成功标准
- 优先使用多选题，每轮只问一个问题
- 当合理假设风险较低时，跳过提问；说明假设并继续

#### 停止提问条件

当以下三点都足够清晰时，立即停止提问：

1. 成功的定义
2. 硬性约束
3. 优先级权衡

若已清晰，转入备选方案和建议阶段。

#### 关键问题过滤器（仅当"是"时才问）

提问前，检查所有条目：

1. 这个决策是否会阻碍进展，或若猜错会导致返工？
2. 这个答案能否从现有代码/文档/背景中推导出来？
3. 这个问题是否能消除解决方案设计中的一个有意义的分支？

若任何答案为"否"，则不要提问；做出并记录合理的假设。

#### 推荐提问顺序

需要提问时，按以下顺序：

1. 成功定义（什么结果证明任务完成）
2. 硬性约束（时间、范围、技术、平台、禁止变更）
3. 优先级权衡（速度 vs 质量，灵活性 vs 简洁性）

避免过早询问实现细节。

### 3) 探索备选方案

- 提供自适应数量的备选方案：
  - 明显/低风险任务：1 个方案
  - 普通任务：2 个方案
  - 高不确定性任务：3 个方案
- 以建议为先，并解释原因
- 简要列出风险和权衡

#### 决策矩阵（保持简洁）

用四个因素评估每个方案：

1. 影响（对用户/业务的价值）
2. 成本（时间/复杂度）
3. 风险（失败/退化的不确定性）
4. 可逆性（回滚/变更的难易程度）

### 4) 渐进式呈现方案

覆盖以下领域（根据复杂度调整深度）：

- 目标和成功标准
- 范围和非目标
- 工作流程/流程步骤
- 风险、依赖和边界情况
- 验证方法（如何知道已完成）

每个部分后，询问是否正确，再继续。

对于低复杂度请求，跳过逐节确认，使用一次最终确认。

## 规划质量原则

- 保持单元聚焦，易于推理
- 在单元之间定义清晰的接口
- 避免无关的重构
- 仅在直接有助于当前目标时改善周边结构
- 严格应用 YAGNI 原则

## 视觉辅助规则

若视觉解释可能很有用，以独立回复的形式提供以下准确信息：

"我们正在处理的一些内容，如果能在浏览器中展示给您可能会更容易解释。在进行过程中，我可以制作模型图、流程图、对比图和其他可视化内容。此功能仍处于早期阶段，可能会消耗较多 token。是否愿意尝试？（需要打开本地 URL）"

不要将此邀请与其他内容混合。等待用户回复后再继续。

仅在决策明显需要视觉呈现时（布局、流程、视觉层次）提供此邀请，否则跳过。

## 文档

当不确定性或影响较高且需要完整方案时：

1. 将方案文档写入 `docs/YYYY-MM-DD-<topic>-plan.md`
2. 确保方案清晰、简洁且可执行

建议章节：

- 问题与目标
- 非目标
- 约束
- 提议的方法
- 已考虑的备选方案
- 风险与缓解措施
- 验证和验收标准
- 发布说明（如需要）

## 自我审查循环（无子代理）

由于不支持子代理，在请求用户最终审查前，执行以下内部审查循环：

1. 从头到尾通读完整方案
2. 检查歧义、矛盾和缺失的决策
3. 检查权衡和边界是否明确
4. 检查验证方法是否具体到可执行
5. 修订并重复，最多 3 轮（低复杂度任务使用更少轮次）

若 3 轮后仍不确定，明确向用户提出不确定性。

## 用户审查关卡

自我审查后，询问：

"方案已写入 `<path>`。请审阅，并告知是否需要在执行前进行任何修改。"

若用户要求修改，更新方案并再次进行自我审查。

## 批准后的过渡

用户批准方案后：

- 仅在获得批准后才开始执行
- 严格按照批准的方案执行

## 最终规则

先规划，后执行。

目标不是快速采取第一个行动，而是减少走弯路，获得更高质量的结果。

少问，决策更好：只问会改变方案的问题。

头脑风暴始终以一句清晰的建议句结束。
