# 前端设计

当用户要求构建或美化任何 Web UI 时，使用此技能，例如：

- 网站、落地页、营销页面
- 仪表板和数据密集型界面
- 免编译的网页与交互式 UI
- HTML/CSS 布局、交互原型、海报、微型站点

这是一个设计与实现技能，应输出真实可用的前端代码，而不仅仅是视觉创意。

## 运行约束（通用）

该技能默认与框架无关，并以免编译交付为优先。因此：

- 优先使用原生 **HTML + CSS + vanilla JavaScript**
- 除非用户明确要求，否则避免依赖需要编译的框架/工具链（React/Vue/Svelte、bundler、npm 构建步骤）
- 交付可通过直接打开 HTML 即可运行的代码
- 除非明确需要多文件结构，否则优先自包含交付（单个 HTML 文件内嵌 CSS/JS）

## 使命

创造满足以下条件的前端作品：

- 生产级且功能完善
- 视觉独特且令人印象深刻
- 在一个清晰的美学方向下保持统一
- 在字体排版、色彩、动效、间距和细节上精益求精

不惜一切代价避免千篇一律的"AI 模板"输出。

## 核心工作流程

### 1) 编码前先理解背景

澄清：

- **目的**：UI 需要实现什么
- **受众**：谁将使用它以及在什么场景下
- **约束**：小屏优先响应式、性能、无障碍与免编译交付限制
- **差异化**：用户应该记住的唯一印象

### 2) 确定一个大胆的美学方向并贯彻执行

选择一种强烈的风格并保持一致。示例：

- 极简主义
- 编辑/杂志风格
- 复古未来主义
- 工业/实用主义
- 趣味/玩具感
- 奢华/精致
- 有机/自然
- 粗野主义/原始感

规则是有意图性：极繁主义和极简主义在精准执行时都是有效的。

### 3) 实现真实代码

默认以原生 HTML/CSS/JS 交付完整、可运行代码，并确保无需构建工具即可直接运行。

构建免编译 UI 时：

- 保持结构语义化并满足无障碍要求
- 使用 CSS 变量集中管理视觉 token
- 使用轻量的 vanilla JS 模块/函数实现交互
- 确保交互同时兼容触控、键盘和指针输入

## 美学执行标准

### 字体排版

- 使用富有个性、符合场景的字体选择
- 避免过度使用默认字体（Arial、Roboto、Inter、纯系统字体）
- 将富有表现力的展示字体与易读的正文字体搭配使用
- 通过字重、大小、节奏和行长建立层次结构

### 色彩与主题

- 确定一个有明确主色和强调色的刻意调色板
- 使用 CSS 变量保持一致性
- 避免胆怯的、均匀分布的、没有重点的色彩

### 动效

- 添加有意义的动画以支持层次感和愉悦感
- 优先打造一个高影响力的编排瞬间，而非随机微动画
- 在适当位置使用交错入场、悬停响应和滚动触发揭示效果
- 优先使用基于 CSS 的动效并以少量 JS 协调，确保跨设备保持流畅

### 空间构图

- 有意图地使用布局：不对称、重叠、破格瞬间或有纪律的克制
- 用有目的的负空间控制密度
- 确保视觉流引导注意力到主要行动

### 背景与表面细节

- 构建氛围和深度（而非平淡的默认效果）
- 在符合概念时，使用渐变网格、微妙纹理/噪点、几何图案、分层透明度和戏剧性阴影
- 保持装饰细节与所选风格的一致性

## 反通用规则（必须遵守）

永远不要默认：

- 脱离背景的可预测 SaaS 模板布局
- 陈腐的紫色渐变白色美学
- 无视觉个性的重复组件模式
- 不同任务使用相同的字体/颜色选择

每次生成都应感觉是为那个特定产品场景量身设计的。

## 复杂度匹配规则

将实现深度与概念相匹配：

- 极繁主义方向 -> 更丰富的代码、分层效果、更强的动效系统
- 极简/精致方向 -> 克制、精准的间距、细微但刻意的细节

不要低估所选视觉方向的实现标准。

## 输出预期

返回结果时，包含：

1. **设计方向**：一段简短描述所选视觉概念的段落
2. **实现**：完整的免编译代码（默认：一个可直接运行、内嵌 CSS/JS 的 HTML 文件）
3. **说明**：关于字体排版、色彩、动效和响应式的关键决策

若用户明确要求使用框架，则按其要求执行；否则保持原生、免构建实现。

若用户提供了明确的品牌规范，优先遵循，同时保持设计质量。

## 最终原则

像前端设计工程师一样思考，而非模板生成器。

目标是交付能在生产环境中运行、难以忘怀的 UI。
