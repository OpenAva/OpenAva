# 自我进化专家

当以下情况适用时，使用本技能：

- **多步骤长周期任务**（编程、数据分析、深度研究报告等），需要在执行中动态优化策略，并将经验延续到未来任务。
- **高风险任务**（生产环境变更、合规检查等），需要内置"批评者审查"环节，在交付前强制自校正。
- **重复性任务**，需要记住用户偏好、领域 SOP 或过往失败模式，并在下次自动复用。

## 输入

- `task_goal`：清晰、可衡量的任务成功标准。
- `evaluation_criteria`：评估方式——自动化测试、数据规则或人工审查清单。
- `context`（可选）：相关历史会话记录、用户画像或领域参考文档。

## 核心工作流

按顺序执行每个阶段。任务完成后，**反思**与**写入记忆**阶段不可跳过。

### 阶段 1 — 行动

基于当前任务目标及从记忆中注入的经验，规划并执行任务。

产出：
- 任务最终输出。
- 执行日志，包含关键决策与支撑证据。

### 阶段 2 — 评估

将实际产出与 `evaluation_criteria` 进行对比。

产出包含以下内容的结构化评估报告：
- 通过/失败判定与评分。
- 已识别的失败点及支撑证据。
- 已识别的成功要素。

### 阶段 3 — 反思

将评估结果提炼为一条或多条纯文本**经验**——每条对应一个独立的洞察。

规则：
- 只有存在可验证证据（日志、测试结果、数据快照）时，才记录经验。无证据支撑的结论必须标记 `[confidence:low]`。
- 每条经验控制在 1–3 句话：触发条件、出错原因（如有），以及下次该做什么。
- 用简短的领域标签标记每条经验，便于未来检索。

经验格式：
```
[lesson][标签1/标签2][confidence:high] 当 <触发条件> 时，执行 <操作>。失败根因（如有）：<原因>。
```

示例：
```
[lesson][golang/concurrency][confidence:high] 生成 Go 并发测试时，始终加入 -race 标志并使用 t.Parallel() 执行子测试。失败根因：缺少 -race 导致竞态条件未被检出。
```

### 阶段 4 — 写入记忆

将每条经验写入合适的记忆层级。不新建任何额外文件。

- **`HISTORY.md`**（通过 `memory_append_history`）：将所有任务级经验写入此处，附带领域标签。这是经验的主要存储——按需检索，不会在每次会话中全量加载。
- **`MEMORY.md`**（通过 `memory_write_long_term`，mode: append）：只在经验真正跨领域通用、且值得在每次会话中注入时，才写入此处。保持克制——`MEMORY.md` 始终全量注入到 System Prompt，必须保持精简。

### 阶段 5 — 进化（下一任务启动前）

在执行新任务之前，从记忆中检索相关经验并注入。

检索规则：
- 调用 `memory_history_search`，以当前任务的领域和类型为关键词或正则进行检索。
- 优先采用 `[confidence:high]` 的经验；`[confidence:low]` 的条目仅作参考。
- 注入前对重叠经验去重。

注入规则：
- 将检索结果改写为**提醒**的自然语言形式——简短祈使句，不暴露原始标签文本。
- 保持注入简洁：目标不超过 5 条。

注入示例：
```
来自过往经验的提醒：
- 编译 Go 并发测试时，始终加入 -race 标志。
- 使用相对路径，禁止硬编码绝对路径。
- 在返回结果前，先对输出结构进行合约校验。
```

## 批评者循环（高风险任务）

对于标记为高风险或高不确定性的任务，在阶段 1 之后额外执行一轮审查：

1. 依据预设原则或安全底线，对初步产出进行评估。
2. 列出具体违规点或缺口，并附支撑证据。
3. 针对每个问题对产出进行改写或修补。
4. 重新评估后，再进入阶段 2。

## 质量控制

- **置信度分级**：High / Medium / Low。缺乏可验证证据的结论必须降级。检索时对低置信度经验降权使用。
- **证据优先**：每条失败原因与经验规则，必须可追溯到具体产物——日志行、测试结果或数据样本。禁止凭空推测。
- **防止错误固化**：未经重新验证的低置信度经验，不得作为默认规则使用。

## 进化度量（长期追踪）

通过以下指标确认本技能持续生效：
- 同类任务的失败率随时间下降。
- 同类问题从发现到修复的平均耗时减少。
- 完成特定任务所需的平均迭代次数减少。
