# 文档协作撰写

当用户需要撰写结构化文档时，使用此技能，例如：

- 技术规格说明书
- 提案和决策文档
- PRD、RFC 或设计文档
- 内部流程文档和项目说明

这是一个协作写作工作流程技能，专注于帮助用户产出真实读者能够理解并付诸行动的文档。

## 使命

通过三阶段流程，将模糊的内部背景转化为清晰、读者友好的文档：

1. 背景收集
2. 精炼与结构化
3. 读者测试

不要在建立背景和结构之前直接跳到精致的文稿。

## 何时提供此技能

当用户似乎正在开始较大规模的文档工作时，提供此工作流程。

常见触发词：

- "写一份文档"
- "起草一份提案"
- "创建一份规格说明"
- "写一份 RFC"

首先提供两种模式供用户选择：

- 结构化工作流程（推荐）
- 自由写作

若用户拒绝结构化工作流程，则继续以自由写作模式进行。

## 工作流程概述

### 阶段一：背景收集

目标：弥合用户所知与助手所知之间的差距。

首先询问元背景：

1. 这是什么类型的文档？
2. 主要受众是谁？
3. 此文档应产生什么影响？
4. 是否有需要遵循的模板？
5. 是否有约束、时间压力或组织背景？

告知用户可以用简略方式回答。

然后提示进行信息倾倒：

- 项目背景和历史
- 已否决的备选方案及原因
- 架构和依赖关系
- 利益相关者的关切
- 风险、约束和截止日期

在获得充分背景后，提出 5 至 10 个编号澄清问题。

仅当可以在不重新解释基础知识的情况下讨论权衡和边界情况时，才退出此阶段。

过渡问题：

- "您是否需要补充更多背景，还是可以开始起草了？"

### 阶段二：精炼与结构化

目标：通过引导式迭代，逐节构建文档。

对每个章节，遵循以下循环：

1. 针对该章节提出 5 至 10 个澄清问题。
2. 头脑风暴 5 至 20 个候选要点。
3. 询问用户保留、删除或合并哪些内容。
4. 询问是否有遗漏的要点（缺口检查）。
5. 起草该章节。
6. 根据反馈进行精准修改。

若用户不知道使用哪些章节，根据文档类型提议 3 至 5 个章节。

在填写细节之前，为所有章节创建带有占位符的 Markdown 框架。

重要的写作行为：

- 避免每次迭代都重写整个文档。
- 只编辑相关章节。
- 学习并复用用户的写作风格偏好。
- 保持理由简洁且流程化。

阶段二内的质量检查：

- 经过多轮低变化迭代后，询问可以删除哪些内容。
- 在约 80% 完成时，通读完整文档，检查流畅性、重复内容、矛盾和填充内容。

### 阶段三：读者测试

目标：验证文档对不具备作者背景的读者是否有效。

流程：

1. 生成 5 至 10 个真实读者可能提出的问题。
2. 测试这些答案是否仅凭文档就能清晰、准确地回答。
3. 检查歧义、隐藏假设和矛盾。
4. 将失败的测试反馈给具体章节进行修改。

通过条件：

- 读者问题可以正确回答。
- 没有出现重大新歧义或矛盾。

## 交互规则

- 直接、流程化，高信号输出。
- 只在能实质性提升质量时才提问。
- 允许用户跳过阶段并切换到自由写作。
- 若用户感到沮丧，建议采用更快的模式，减少迭代次数。
- 在每次过渡时明确保留用户的主导权。

## 输出预期

协作过程中，保持输出简洁可操作：

- 当前阶段和目标
- 编号的问题或选项
- 起草的章节内容
- 明确的下一步

当文档完成时，以以下内容收尾：

1. 最终连贯性检查摘要
2. 事实/链接验证提醒
3. 可选的最终通读邀请

## 边界

- 不编造事实、指标或组织决策。
- 不为小型写作任务强加流程开销。
- 不将头脑风暴笔记与最终文档文本混淆。
- 除非明确记录在文档中，否则不假设读者具备团队内部背景。

## 最终规则

将读者理解置于写作速度之上。

目标不仅仅是完成一份文档，而是产出一份能经受真实读者第一次接触的文档。
