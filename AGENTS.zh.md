# Agent 工作流程与约束

## 必须遵循的流程

### 1. 澄清任务

- 必须先识别真实目标，再提出方案或修改实现。
- 必须先识别硬约束、明确需求与非目标。
- 必须先识别核心权衡，并基于当前任务做选择，而不是基于想象中的未来需求做选择。

### 2. 设计最简充分方案

- 必须从第一性原理出发。
- 必须选择能够完整解决任务的最简单方案。
- 必须优先复用现有代码、类型与流程，而不是新增实体。
- 必须让结构清晰、命名直白，优先于炫技。

### 3. 直接实现

- 必须保持数据流显式、归属清晰。
- 必须避免预防性抽象、隐藏间接层与一次性包装。
- 除非解决任务确有必要，否则不得新增参数、层级、状态或辅助结构。
- 必须删除不再服务当前设计的代码。

### 4. 在可用之后再简化一次

- 第一版可用后，必须再做一次简化。
- 必须去掉不必要的分支、参数、状态、层级与抽象。
- 当合并重复能直接提升清晰度时，必须合并重复。
- 必须收紧边界，并修正所有误导性命名。

### 5. 验证完成

- 必须验证真实需求已在实际行为中成立，而不只是理论上成立。
- 必须验证最终设计易于解释。
- 必须验证没有保留非必要实体。
- 必须验证所有保留下来的复杂度都确实由任务需要支撑。

## 硬性约束

- 不得为想象中的未来需求做设计。
- 不得引入没有明确当前价值的实体。
- 不得以“已经能运行”为停止条件；如果还可以继续简化，就不能停止。
- 不得保留死代码、误导性命名或没有理由的复杂度。

## 决策标准

当存在多个可行方案时，必须按以下顺序选择：

1. 最符合真实需求的方案。
2. 结构与推理路径最简单的方案。
3. 最容易阅读和解释的方案。
4. 不依赖额外抽象、最容易维护的方案。
